Designing a High-Throughput Validation Engine

In FinTech, every loan application, every KYC document, and every transaction payload must pass through a gauntlet of validation rules before it can proceed. At Credit Saison, we needed a validation engine that could handle millions of documents daily with sub-50ms latency and near-zero error rates. This is the story of how we designed it.

The Problem Space

Our initial validation layer was a monolithic Spring Boot service with hardcoded rules scattered across dozens of service classes. Adding a new validation rule meant a code change, a PR, a deployment. Worse, the rules weren't composable — each product team maintained their own validation logic, leading to drift, duplication, and inconsistency.

We needed a system that was: declarative (rules as configuration, not code), extensible (custom keywords for domain logic), fast (sub-50ms P95 for complex schemas), and observable (clear error paths for every validation failure).

Architecture Decisions

We chose JSON Schema as our validation language. It's a well-defined standard, has excellent tooling, and — crucially — it separates the "what" (the schema) from the "how" (the engine). This let product teams define their own schemas without touching engine code.

The engine itself is built on NetworkNT's json-schema-validator, which we extended heavily. We added custom keywords for Indian financial domain logic: PAN validation, Aadhaar checksum verification, IFSC code format checking, and cross-field conditional rules (e.g., "if employment type is salaried, salary slip is mandatory").

Performance Engineering

The naive approach — compile schema, validate document, return — added 15-30ms of overhead per request just for schema compilation. Our key insight: schemas change rarely but documents flow constantly. We introduced a schema compilation cache with versioned invalidation. Compiled schemas are held in memory and only recompiled when the schema version changes.

This single optimization dropped our P95 from 45ms to 8ms for cached schemas. For cold starts, we added a warm-up routine that pre-compiles the 50 most-used schemas at service startup.

For the Go rewrite of the hot path, we leveraged goroutines to validate independent sub-schemas concurrently. A loan application with 12 document types can have all 12 validated in parallel, with results merged at the end. This brought our P99 down by another 40%.

Error Reporting

A validation engine is only as good as its error messages. Generic "validation failed" responses are useless to frontend teams and operations staff. We built a custom error formatter that translates JSON Schema validation paths into human-readable, localizable error messages. Each error carries a unique code, the field path, the expected value, and the actual value.

Lessons Learned

Three key takeaways from this project:

1. Invest in schema tooling early. We built a schema editor UI, a test harness for running documents against schemas, and a diff tool for comparing schema versions. These tools paid for themselves within weeks.

2. Measure everything. We instrument every validation with latency histograms, error rate counters, and schema hit/miss ratios. When a new schema causes a latency spike, we know within minutes.

3. Design for the next 10x. Our original design supported 100K validations/day. Today we process 2M+ daily. The schema-as-configuration approach meant scaling required zero code changes — just more compute.

The validation engine is now one of the most stable and highest-throughput services in our platform, processing every financial decision with confidence.
