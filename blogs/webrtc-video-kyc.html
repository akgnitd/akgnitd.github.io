Building a WebRTC SFU for Video KYC

Video KYC (Know Your Customer) became a regulatory requirement for Indian financial institutions after the RBI's 2020 amendment to the Master Direction on KYC. When we set out to build our VKYC platform, we knew that a naive peer-to-peer WebRTC implementation would collapse under the weight of enterprise networks, compliance recording mandates, and the sheer variability of Indian mobile connectivity. This post is a deep technical walkthrough of the Mediasoup-based Selective Forwarding Unit (SFU) we built, the queue service that manages agent assignment, and every hard-won lesson from shipping this to production.

Why an SFU Over Peer-to-Peer

A Video KYC session is not a casual video call. It involves three distinct participants with asymmetric roles: the customer (on a mobile phone, often on patchy 3G), the agent (on a desktop inside a corporate network), and an optional auditor who observes the entire session without sending any media. In a peer-to-peer mesh topology, each participant must send their media stream to every other participant. With three people, that means six separate media streams traversing the network. Add a fourth "ghost" peer for compliance recording and you have a mesh that no mobile phone on Jio's congested towers can sustain.

An SFU fundamentally changes the architecture. Each participant sends their media once to the server. The server then selectively forwards those streams to the participants who need them. The customer uploads one audio and one video track. The agent receives those tracks and uploads their own. The auditor receives both but uploads nothing. The recording pipeline taps directly into the server-side media without adding any load on the participants' devices or network connections.

We evaluated three SFU frameworks before committing: Janus (C-based, excellent performance but the plugin architecture made custom logic painful), Jitsi Videobridge (Java-based, optimized for large conferences but heavyweight for our 2-3 person sessions), and Mediasoup (C++ media workers controlled by a Node.js signalling layer). Mediasoup won because its API design maps cleanly to our domain model: a Router is a session, a Transport is a participant's network connection, a Producer is an outgoing track, and a Consumer is an incoming track. The Node.js layer let us write signalling logic in the same language as our backend, and the C++ workers handle the actual media routing at wire speed.

Mediasoup Configuration: The Production Config

Here is the actual Mediasoup configuration from our VKYC service. Every value in this config was tuned through months of production traffic:

export const config = {
    worker: {
        rtcMinPort: parseInt(process.env.RTC_MIN_PORT || '40000', 10),
        rtcMaxPort: parseInt(process.env.RTC_MAX_PORT || '40100', 10),
    },
    mediaCodecs: [
        {
            kind: 'audio',
            mimeType: 'audio/opus',
            clockRate: 48000,
            channels: 2,
        },
        {
            kind: 'video',
            mimeType: 'video/VP8',
            clockRate: 90000,
            parameters: {
                'x-google-start-bitrate': 1000,
            },
        },
        {
            kind: 'video',
            mimeType: 'video/H264',
            clockRate: 90000,
            parameters: {
                'packetization-mode': 1,
                'profile-level-id': '42e01f',
            },
        },
    ],
    webRtcTransport: {
        maxIncomingBitrate: 1_500_000,
        initialAvailableOutgoingBitrate: 1_000_000,
    },
};

The port range 40000-40100 gives us 100 UDP ports for RTP traffic. Each WebRTC transport uses one port, and a typical session with three participants needs six transports (send + receive per participant), so this range supports about 16 concurrent sessions per worker process. We run multiple workers and scale horizontally by adding instances behind our load balancer.

We declare both VP8 and H264 codecs. VP8 is the default for Chrome-to-Chrome sessions and offers better simulcast support. H264 with profile 42e01f (Constrained Baseline, Level 3.1) is the fallback for Safari and iOS devices where VP8 encoding produces green artifacts and hardware acceleration is only available for H264. The x-google-start-bitrate of 1000 kbps tells Chrome's bandwidth estimator to start at a reasonable rate rather than ramping up slowly from zero, which is critical for KYC sessions where the first 10 seconds matter.

The transport bitrate caps are deliberate. maxIncomingBitrate at 1.5 Mbps prevents a single participant on a fast connection from overwhelming the server. initialAvailableOutgoingBitrate at 1 Mbps gives the bandwidth estimator a healthy starting point for outbound streams. These values were calibrated against real Indian network conditions: a customer on Airtel 4G in a tier-2 city typically sustains 2-3 Mbps, but during peak hours that drops to 800 kbps-1.2 Mbps.

The Queue Service: Agent Assignment

Before any media flows, the customer must be connected with an available agent. We built a priority-based queue service backed by Redis sorted sets. When a customer initiates a VKYC session, the backend calls the queue service:

async enqueueSession(session) {
    const score = (4 - session.priority) * 1_000_000 + nowEpoch;
    await this.redis.zadd(queueKey, score, JSON.stringify(entry));
    await this.tryAssignNext(session.tenantId);
    this.gateway.notifySessionUpdate(
        session.id,
        'queue:position',
        { position, estimatedWait }
    );
}

The scoring formula (4 - priority) * 1_000_000 + nowEpoch is a compound sort key. Priority is 1-4 (1 = highest), so subtracting from 4 inverts it. Multiplying by one million creates a band wide enough that all sessions of priority N sort before all sessions of priority N+1, while the nowEpoch component provides FIFO ordering within the same priority band. A priority-1 session submitted at epoch 1700000000 gets score 3000000 + 1700000000 = 1703000000, while a priority-3 session at the same time gets 1000000 + 1700000000 = 1701000000. Redis ZRANGEBYSCORE gives us the next session to assign in O(log N).

The tryAssignNext method checks the pool of available agents for the tenant. Agent availability is tracked via a Redis set that gets updated on WebSocket heartbeats. When an agent is matched, the session transitions from WAITING to AGENT_ASSIGNED, and both the customer and agent receive WebSocket notifications to begin the media connection handshake. The estimatedWait is calculated from the average assignment time over the last 50 sessions for that tenant, which gives customers a realistic expectation rather than a meaningless spinner.

Multi-Party Session Architecture

Each KYC session maps to a Mediasoup Router, which acts as an isolated media routing context. The Router holds the codec capabilities and manages all transports within the session. When a participant joins, the signalling server executes the following sequence:

First, it creates two WebRtcTransports for the participant: one for sending media and one for receiving. Each transport gets its own ICE candidates, DTLS parameters, and SCTP parameters (for data channels). The transport creation returns these parameters to the client, which uses them to configure its local RTCPeerConnection.

Second, the client connects its send transport and starts producing media. For the customer, this means one audio Producer (microphone) and one video Producer (camera). For the agent, the same. The auditor creates no Producers since they only observe.

Third, for each existing Producer in the Router, the signalling server creates a Consumer on the new participant's receive transport. The customer gets Consumers for the agent's audio and video. The agent gets Consumers for the customer's audio and video. The auditor gets Consumers for everyone's audio and video. This is where the SFU's selective forwarding shines: the server decides who receives what, and participants with different roles get different sets of Consumers.

The signalling server maintains a Map<sessionId, SessionState> where SessionState tracks all participants, their transports, producers, and consumers. When a participant disconnects, the server closes their transports (which automatically closes all associated producers and consumers), removes them from the session state, and notifies other participants. If the disconnected participant was the customer, the agent sees a reconnection timer. If it was the agent, the session goes into ON_HOLD and the queue service tries to assign a new agent.

Adaptive Bitrate for Indian 3G/4G Networks

Indian mobile networks present a unique challenge. A customer in Mumbai might be on a strong Jio 4G connection with 15 Mbps bandwidth, while a customer in a rural area of Uttar Pradesh might be on BSNL 3G with barely 400 kbps. Even within a single session, bandwidth can swing wildly as the customer moves between cells or as tower congestion changes.

We handle this with simulcast on the video Producer. The customer's browser encodes three spatial layers simultaneously: high (720p, ~1200 kbps), medium (360p, ~500 kbps), and low (180p, ~150 kbps). The SFU monitors the receiver's available bandwidth through RTCP receiver reports and dynamically selects which layer to forward.

The layer selection is role-aware. The agent's Consumer is configured with preferredLayers: { spatialLayer: 2, temporalLayer: 2 } (highest quality) because they need to read document text and verify facial features. The auditor's Consumer uses preferredLayers: { spatialLayer: 1, temporalLayer: 1 } (medium quality) since they only need to confirm the session is proceeding correctly. When the customer's uplink bandwidth drops, the SFU automatically downgrades to a lower spatial layer. The customer sees no "reconnecting" UI, the video just becomes slightly less sharp for a few seconds until bandwidth recovers.

We also implemented a custom bandwidth probing mechanism. When a session starts, the SFU sends padding packets to estimate the available bandwidth before the first video keyframe arrives. This eliminates the "black screen for 3 seconds while bandwidth ramps up" problem that plagues naive WebRTC implementations. On our production traffic, median time-to-first-frame dropped from 2.8 seconds to 0.9 seconds after implementing probing.

FFmpeg Recording Pipeline for Compliance

Every Video KYC session must be recorded end-to-end and archived for a minimum of 8 years per RBI guidelines. We built the recording pipeline using Mediasoup's PlainTransport, which pipes raw RTP packets to a local UDP port. An FFmpeg process listens on that port and muxes the streams into an MP4 container.

The FFmpeg command for a typical session recording looks like this:

ffmpeg -protocol_whitelist pipe,udp,rtp \
  -i customer_video.sdp \
  -i customer_audio.sdp \
  -i agent_video.sdp \
  -i agent_audio.sdp \
  -filter_complex "[0:v][2:v]hstack=inputs=2[v]; \
                    [1:a][3:a]amix=inputs=2[a]" \
  -map "[v]" -map "[a]" \
  -c:v libx264 -preset veryfast -crf 23 \
  -c:a aac -b:a 128k \
  -movflags +faststart \
  -f mp4 /recordings/{sessionId}.mp4

The hstack filter places the customer and agent video side-by-side in a single frame. The amix filter combines both audio tracks. The -preset veryfast keeps CPU usage reasonable during real-time encoding, and -crf 23 gives us good quality at a manageable file size. The -movflags +faststart moves the moov atom to the beginning of the file so auditors can start playback before the entire file downloads from S3.

The hardest part was audio-video synchronization. Audio arrives as Opus packets at 48kHz clock rate, video arrives as VP8 or H264 packets at 90kHz clock rate, and each stream has its own RTP timestamp sequence. We generate SDP files dynamically for each recording session with the correct payload types and clock rates, and rely on FFmpeg's internal timestamp handling to maintain lip sync. In testing, we measured sync drift under 40ms across 30-minute sessions, which is well within the perceptible threshold of ~80ms.

Once the session ends, the recording is uploaded to S3 with AES-256 server-side encryption, tagged with the session ID, tenant ID, timestamp, and agent ID. A separate archival pipeline moves recordings older than 30 days to S3 Glacier Deep Archive, reducing storage costs from $0.023/GB/month to $0.00099/GB/month. For our volume of ~200 sessions per day averaging 8 minutes each, the storage cost dropped from ~$180/month to ~$8/month.

CoTURN for Enterprise NAT Traversal

WebRTC's ICE framework tries multiple connectivity paths in parallel: host candidates (direct LAN), server-reflexive candidates (via STUN to discover the public IP), and relay candidates (via TURN when nothing else works). In our production traffic, the connectivity breakdown is roughly: 45% direct (customer on mobile + agent on cloud desktop), 40% STUN (standard NAT traversal), and 15% TURN relay.

That 15% is almost entirely from agent desktops inside corporate networks. Banks and NBFCs run symmetric NATs, deep packet inspection firewalls that block non-HTTP UDP traffic, and sometimes explicit proxies that only allow TCP on ports 80 and 443. For these environments, TURN relay is the only option.

We deploy CoTURN on dedicated high-bandwidth instances (c5n.xlarge on AWS, which provides 25 Gbps network bandwidth). CoTURN is configured to listen on both UDP (3478) and TCP (443) with TLS, because some corporate firewalls only allow outbound TCP on 443. The TURN credentials are short-lived: our backend generates a username/password pair with a 5-minute TTL for each session, using the TURN REST API's shared-secret mechanism. This prevents credential reuse and makes it impossible for a compromised client to relay arbitrary traffic through our TURN server.

# CoTURN configuration excerpt
listening-port=3478
tls-listening-port=443
min-port=49152
max-port=65535
use-auth-secret
static-auth-secret=${SHARED_SECRET}
realm=vkyc.example.com
total-quota=100
bps-capacity=0
stale-nonce=600
no-multicast-peers

The total-quota=100 limits concurrent relay allocations per user, preventing a single misbehaving client from exhausting the server. stale-nonce=600 forces credential refresh every 10 minutes for long sessions. no-multicast-peers prevents the TURN server from being used as a multicast relay, which is a known attack vector.

The 19-State Session Lifecycle

A Video KYC session is a compliance workflow, not a casual video call. We modelled it as a 19-state finite state machine where every transition is logged, validated, and triggers specific side effects. Here is the complete state diagram:

CREATED -> SCHEDULED -> WAITING -> AGENT_ASSIGNED -> CONNECTING
    -> CONNECTED -> DOC_CAPTURE -> LIVENESS_CHECK -> OTP_SENT
    -> OTP_VERIFIED -> AGENT_REVIEW -> APPROVED | REJECTED

// Interruption states (reachable from multiple states):
CONNECTED -> ON_HOLD -> CONNECTED
CONNECTED -> RECONNECTING -> CONNECTED | CUSTOMER_DROPPED | AGENT_DROPPED
ANY -> TIMED_OUT
ANY -> CANCELLED
APPROVED | REJECTED -> ARCHIVED

Each transition fires a set of side effects. When the state moves from WAITING to AGENT_ASSIGNED, the queue service removes the session from the Redis sorted set and marks the agent as busy. When CONNECTING transitions to CONNECTED, the recording pipeline starts. When CONNECTED moves to DOC_CAPTURE, the SFU switches the customer's video Consumer on the agent's side to the highest spatial layer and the frontend enables the document capture overlay. When the state reaches APPROVED or REJECTED, the recording pipeline stops, the media transports are closed, and an audit log entry is written with the agent's verdict, the document hashes, and the liveness check results.

The RECONNECTING state handles transient network drops. When the customer's WebSocket disconnects, the session moves to RECONNECTING with a 60-second timer. If the customer reconnects within that window, they rejoin the same session with new transports but the same session ID. If the timer expires, the state moves to CUSTOMER_DROPPED and the agent can either wait for a manual rejoin or close the session. This pattern saved roughly 12% of sessions that would otherwise have been marked as failed due to brief network interruptions.

Browser Quirks and Hard Lessons

Shipping WebRTC to production across every browser and device combination taught us more about browser inconsistencies than any other project.

Safari VP8 green frames: Safari on iOS 15 and below has a bug where VP8-encoded video occasionally produces bright green frames, especially after a keyframe request during simulcast layer switching. The root cause is a race condition in Safari's VP8 decoder when the spatial layer changes mid-GOP. Our fix was to negotiate H264 as the preferred codec whenever the User-Agent indicates Safari, using the profile-level-id: 42e01f (Constrained Baseline) which has hardware acceleration on all iOS devices since the iPhone 6.

Background tab throttling: When a customer switches to their SMS app to read an OTP (which happens in every single KYC session), both Chrome and Safari aggressively throttle or suspend the background tab. Chrome reduces timer resolution to 1 second and may pause the WebRTC connection entirely after 5 minutes. Safari on iOS is more aggressive and suspends the tab almost immediately. We detect this via a heartbeat: the client sends a ping every 3 seconds, and if the server misses 5 consecutive pings, it marks the participant as "temporarily away" rather than disconnected. The agent sees a visual indicator, and when the customer returns, the connection resumes without requiring a fresh ICE negotiation. We achieve this by keeping the DTLS session alive and only resuming the media flow.

getUserMedia permission persistence: On Android Chrome, the camera permission prompt sometimes reappears after a page navigation within a single-page app. This is caused by the MediaStream being garbage-collected when the React component unmounts. We solved this by holding a reference to the active MediaStream in a global store and reusing it across component mounts rather than calling getUserMedia() again.

Echo cancellation on speakerphone: When the customer uses speakerphone (common during document scanning), the agent hears severe echo. The browser's built-in AEC (Acoustic Echo Cancellation) works well for earpiece and headphone modes but struggles with loudspeaker audio. We mitigate this by applying a server-side audio processing pipeline: the customer's audio Consumer goes through a noise gate that suppresses audio below -40dB (catching most echo) and a simple high-pass filter at 100Hz that removes the low-frequency rumble that AEC typically misses.

Hardware encoder saturation on low-end devices: Budget Android phones (Redmi, Realme) with MediaTek chipsets sometimes fail to produce video frames when the hardware encoder saturates. The browser does not throw an error; the video track simply stops producing frames while track.readyState still reports "live." We detect this by monitoring the RTCOutboundRtpStreamStats.framesEncoded counter: if it stops incrementing for 5 seconds, we force a track restart by calling track.stop() followed by a fresh getUserMedia() and producer.replaceTrack(). This auto-recovery handles the issue transparently in roughly 95% of cases.

Production Numbers and Lessons Learned

After 14 months in production, the system has processed over 85,000 VKYC sessions across 6 tenants. Median session duration is 7 minutes 20 seconds. Connection success rate (customer connects to agent with bidirectional media within 10 seconds) is 97.3%. The remaining 2.7% are almost entirely attributable to customers on extremely congested networks (sub-200 kbps) or devices with broken WebRTC implementations (certain Samsung Internet browser versions).

Building a production Video KYC system taught us that WebRTC is 20% media engineering and 80% edge case handling. The SFU architecture gave us the flexibility to handle compliance requirements without compromising call quality. The queue service ensured fair agent assignment under load. And Mediasoup proved to be a reliable, well-documented foundation that let us focus on our domain logic rather than fighting the media stack.
