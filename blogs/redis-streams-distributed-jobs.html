Redis Streams for Distributed Job Processing

Across three different production services, we use Redis for background job processing -- but each service uses a different Redis consumption pattern based on its specific requirements. The VKYC service (Go) uses Redis Streams with XREADGROUP and XAUTOCLAIM for document OCR processing. The engage-flow service (Java Spring) uses Spring's StreamMessageListenerContainer for workflow event processing. The connector-service (Go) uses the simpler BLPop pattern for webhook delivery. This post is a deep comparison of all three patterns, drawn from real production code, with honest assessments of when each pattern is the right choice.

Why Redis Over Kafka or SQS

Before comparing Redis patterns, it is worth explaining why Redis at all. Kafka is the gold standard for high-throughput event streaming. It handles millions of messages per second with replicated, durable commit logs. But Kafka requires dedicated infrastructure: ZooKeeper (or KRaft), broker clusters, topic partition management, and at least one engineer who understands Kafka operations. For our workload of 10,000-50,000 jobs per day across all three services, Kafka is like renting a freight container to ship a shoebox.

SQS eliminates the operational burden but introduces other limitations: no consumer groups (fan-out requires multiple queues or SNS), no message replay, 256KB message size limit, and you pay per API call. More importantly, SQS adds network latency for every enqueue/dequeue operation since it is a remote AWS service. Our Redis instances are co-located with our application servers, giving us sub-millisecond enqueue/dequeue latency.

Redis was already in our stack for caching, session storage, and rate limiting. Using it for job processing means no new infrastructure, no new operational runbooks, no new monitoring stack. Redis Streams provide append-only log semantics (conceptually similar to Kafka topics), consumer groups for distributed processing, message acknowledgement for at-least-once delivery, and the ability to inspect pending messages and claim stale ones. The trade-off is durability: Redis persistence (RDB snapshots every 60 seconds, AOF with everysec fsync) is not as robust as Kafka's replicated commit log. For our use cases, losing a few in-flight jobs on a Redis crash is acceptable because all our processors are idempotent and we run a reconciliation job that catches gaps.

Pattern 1: Go XREADGROUP + XAUTOCLAIM (VKYC Service)

The VKYC service processes document verification jobs: when a customer uploads an Aadhaar card, PAN card, or bank statement during a Video KYC session, the image is stored in S3 and a job is enqueued to a Redis Stream for OCR extraction and document verification. Here is the core consumption loop from the Go worker:

func (w *Worker) processDocumentJobs(ctx context.Context) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
        }

        // Step 1: Read new messages from the stream
        streams, err := w.redis.XReadGroup(ctx, &redis.XReadGroupArgs{
            Group:    documentConsumerGroup,
            Consumer: w.cfg.consumerName,
            Streams:  []string{documentJobStreamKey, ">"},
            Count:    maxReadCount,
            Block:    readBlockTimeout,
        }).Result()

        if err != nil {
            if err == redis.Nil {
                continue // No new messages, loop back to block again
            }
            w.logger.Error("xreadgroup failed", zap.Error(err))
            time.Sleep(errorBackoff)
            continue
        }

        for _, stream := range streams {
            for _, message := range stream.Messages {
                if err := w.handleDocumentJob(ctx, message); err != nil {
                    w.logger.Error("job processing failed",
                        zap.String("message_id", message.ID),
                        zap.Error(err),
                    )
                    // Don't ACK - message stays in PEL for retry/claim
                    continue
                }

                // ACK on success
                w.redis.XAck(ctx, documentJobStreamKey, documentConsumerGroup, message.ID)
            }
        }
    }
}

The ">" special ID in the Streams argument tells Redis to deliver only messages that have never been delivered to any consumer in this group. Count: maxReadCount (set to 10) limits how many messages we process per iteration, preventing a single worker from grabbing too many jobs during a burst. Block: readBlockTimeout (set to 5 seconds) makes the call block until either messages are available or the timeout expires, avoiding a tight polling loop that would waste CPU.

The critical design decision is on the error path: when handleDocumentJob fails, we do not acknowledge the message. It remains in the consumer's Pending Entries List (PEL). This is the foundation of at-least-once delivery: unacknowledged messages are retried.

Claiming Stale Messages with XAUTOCLAIM

If a worker crashes mid-processing, its pending messages are orphaned in the PEL. No other consumer will receive them because Redis tracks that they were already delivered. This is where XAUTOCLAIM comes in. We run a separate goroutine that periodically claims stale messages:

func (w *Worker) claimStaleMessages(ctx context.Context) {
    ticker := time.NewTicker(staleClaimInterval) // 30 seconds
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            messages, _, err := w.redis.XAutoClaim(ctx, &redis.XAutoClaimArgs{
                Stream:   documentJobStreamKey,
                Group:    documentConsumerGroup,
                Consumer: w.cfg.consumerName,
                MinIdle:  staleMessageThreshold, // 5 minutes
                Count:    maxClaimCount,          // 20
                Start:    "0-0",
            }).Result()

            if err != nil {
                w.logger.Error("xautoclaim failed", zap.Error(err))
                continue
            }

            for _, message := range messages {
                deliveryCount, _ := w.getDeliveryCount(ctx, message.ID)
                if deliveryCount > maxRetries { // maxRetries = 3
                    w.moveToDeadLetter(ctx, message)
                    w.redis.XAck(ctx, documentJobStreamKey, documentConsumerGroup, message.ID)
                    continue
                }

                if err := w.handleDocumentJob(ctx, message); err != nil {
                    w.logger.Warn("stale message reprocessing failed",
                        zap.String("message_id", message.ID),
                        zap.Int("delivery_count", deliveryCount),
                        zap.Error(err),
                    )
                    continue
                }

                w.redis.XAck(ctx, documentJobStreamKey, documentConsumerGroup, message.ID)
            }
        }
    }
}

XAUTOCLAIM is a Redis 6.2+ command that combines XPENDING + XCLAIM into a single atomic operation. It finds messages in the PEL that have been idle for longer than MinIdle (5 minutes) and transfers ownership to the calling consumer. The Start: "0-0" means scan from the beginning of the PEL.

The delivery count check is our dead letter logic. Redis tracks how many times a message has been delivered (incremented each time it is read or claimed). If a message has been delivered more than 3 times and still not acknowledged, it is a poison pill -- probably malformed input that will never process successfully. We move it to a dead letter stream and ACK it from the main stream to prevent infinite retries.

Dead Letter Handling

func (w *Worker) moveToDeadLetter(ctx context.Context, message redis.XMessage) {
    dlqEntry := map[string]interface{}{
        "original_stream":  documentJobStreamKey,
        "original_id":      message.ID,
        "consumer_group":   documentConsumerGroup,
        "failed_consumer":  w.cfg.consumerName,
        "failure_time":     time.Now().UTC().Format(time.RFC3339),
        "payload":          message.Values,
    }

    w.redis.XAdd(ctx, &redis.XAddArgs{
        Stream: documentJobStreamKey + ":dlq",
        Values: dlqEntry,
    })

    w.metrics.IncrCounter("document_jobs_dead_lettered", 1)
    w.logger.Error("message moved to DLQ",
        zap.String("message_id", message.ID),
        zap.Any("payload", message.Values),
    )
}

The dead letter stream (document-jobs:dlq) preserves the original message payload along with diagnostic metadata: which stream it came from, which consumer failed on it, and when. We have an admin API that lists DLQ entries and allows one-click reprocessing (which XADDs the payload back to the original stream).

Pattern 2: Java Spring StreamMessageListenerContainer (engage-flow)

The engage-flow service processes workflow events: when a customer action triggers a marketing flow (e.g., "abandoned cart -> send email after 30 minutes -> if not opened, send push notification"), the event is enqueued to a Redis Stream and processed by Spring's built-in stream listener infrastructure. This pattern is more declarative than the Go approach:

@Configuration
public class RedisStreamConfig {

    @Bean
    public StreamMessageListenerContainer<String, MapRecord<String, String, String>>
            listenerContainer(RedisConnectionFactory connectionFactory,
                              RedisMessageHandler redisMessageHandler) {

        var options = StreamMessageListenerContainer
            .StreamMessageListenerContainerOptions.builder()
            .pollTimeout(Duration.ofSeconds(5))
            .batchSize(10)
            .targetType(MapRecord.class)
            .errorHandler(e -> log.error("Stream listener error", e))
            .build();

        var listenerContainer = StreamMessageListenerContainer
            .create(connectionFactory, options);

        String consumerGroup = appConfig.getConsumerGroup();
        String consumerName = appConfig.getConsumerName();
        String streamKey = appConfig.getStreamKey();

        // Ensure consumer group exists
        try {
            connectionFactory.getConnection().streamCommands()
                .xGroupCreate(streamKey.getBytes(), consumerGroup, ReadOffset.latest(), true);
        } catch (Exception e) {
            // Group already exists - this is fine
        }

        listenerContainer.receive(
            Consumer.from(consumerGroup, consumerName),
            StreamOffset.create(streamKey, ReadOffset.lastConsumed()),
            message -> {
                try {
                    redisMessageHandler.handleRedisMessage(message.getValue());
                    // Auto-ACK on successful processing
                    connectionFactory.getConnection().streamCommands()
                        .xAck(streamKey.getBytes(), consumerGroup,
                               message.getId().getValue().getBytes());
                } catch (Exception e) {
                    log.error("Failed to process message: {}", message.getId(), e);
                    // Don't ACK - message stays in PEL
                }
            }
        );

        listenerContainer.start();
        return listenerContainer;
    }
}

Spring's StreamMessageListenerContainer abstracts away the XREADGROUP loop, the blocking, and the thread management. You configure poll timeout (equivalent to the Block parameter), batch size (equivalent to Count), and provide a message handler lambda. Under the hood, it runs a polling loop in a dedicated thread that calls XREADGROUP with the configured parameters.

The ReadOffset.lastConsumed() tells the container to read from the last consumed position, which translates to the ">" special ID in XREADGROUP. If the consumer restarts, it picks up from where it left off thanks to Redis tracking the group's cursor position.

The trade-off compared to the Go approach is flexibility. Spring's container does not natively support XAUTOCLAIM for stale message recovery. We handle this with a separate scheduled task:

@Scheduled(fixedRate = 60_000) // Every 60 seconds
public void claimStaleMessages() {
    StringRedisSerializer serializer = new StringRedisSerializer();

    // XPENDING to find stale messages
    PendingMessages pending = redisTemplate.opsForStream()
        .pending(streamKey, consumerGroup, Range.unbounded(), 50);

    for (PendingMessage pm : pending) {
        if (pm.getElapsedTimeSinceLastDelivery().compareTo(Duration.ofMinutes(5)) > 0
                && pm.getTotalDeliveryCount() <= 3) {

            // XCLAIM the stale message
            List<MapRecord<String, Object, Object>> claimed =
                redisTemplate.opsForStream().claim(
                    streamKey, consumerGroup, consumerName,
                    Duration.ofMinutes(5), RecordId.of(pm.getIdAsString()));

            for (var record : claimed) {
                try {
                    redisMessageHandler.handleRedisMessage(
                        (Map<String, String>) (Map) record.getValue());
                    redisTemplate.opsForStream()
                        .acknowledge(streamKey, consumerGroup, record.getId());
                } catch (Exception e) {
                    log.error("Stale message reprocessing failed: {}", record.getId(), e);
                }
            }
        } else if (pm.getTotalDeliveryCount() > 3) {
            moveToDeadLetter(pm);
        }
    }
}

Pattern 3: Go BLPop Queue (connector-service)

The connector-service handles webhook delivery to partner systems. Unlike document processing (which requires consumer groups for distributed load balancing), webhook delivery has a simpler requirement: pick up a job, deliver the webhook, mark it done. We use the BLPop pattern, which is Redis's simplest queue primitive:

func (r *RedisQueue) ConsumeJobs(ctx context.Context, queue string, handler JobHandler) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
        }

        result, err := r.RedisClient.BLPop(ctx, 5*time.Second, queue).Result()
        if err != nil {
            if err == redis.Nil {
                continue // Timeout, no messages available
            }
            r.logger.Error("blpop failed", zap.String("queue", queue), zap.Error(err))
            time.Sleep(time.Second)
            continue
        }

        // result[0] is the queue name, result[1] is the payload
        payload := result[1]

        var job WebhookJob
        if err := json.Unmarshal([]byte(payload), &job); err != nil {
            r.logger.Error("invalid job payload",
                zap.String("payload", payload),
                zap.Error(err),
            )
            continue // Malformed payload, skip
        }

        if err := handler.Handle(ctx, &job); err != nil {
            r.logger.Error("job handling failed",
                zap.String("job_id", job.ID),
                zap.Error(err),
            )
            r.handleRetry(ctx, queue, &job, err)
            continue
        }

        r.metrics.IncrCounter("webhooks_delivered", 1)
    }
}

BLPop is a blocking left-pop: it removes and returns the first element of the list, blocking for up to 5 seconds if the list is empty. The producer enqueues jobs with RPush, creating a FIFO queue. This is the simplest possible queue pattern in Redis -- no consumer groups, no message IDs, no ACK mechanism.

The simplicity is both the strength and the limitation. BLPop removes the message from the list atomically, so there is no risk of duplicate delivery (unlike Streams where you must ACK). But if the consumer crashes after popping the message but before completing the webhook delivery, the message is lost. For our webhook use case, this is acceptable because:

1. The webhook job is also recorded in PostgreSQL with a status of "pending." A reconciliation cron job runs every 5 minutes and re-enqueues any job that has been "pending" for longer than 2 minutes. This catches any jobs lost to consumer crashes.

2. Webhook delivery is inherently idempotent from the receiver's perspective (our payloads include a unique event ID that receivers use for deduplication).

Retry with Exponential Backoff

func (r *RedisQueue) handleRetry(ctx context.Context, queue string, job *WebhookJob, jobErr error) {
    job.RetryCount++
    job.LastError = jobErr.Error()
    job.LastAttempt = time.Now().UTC()

    if job.RetryCount > maxRetries { // maxRetries = 5
        r.logger.Error("max retries exceeded, moving to DLQ",
            zap.String("job_id", job.ID),
            zap.Int("retry_count", job.RetryCount),
        )
        r.moveToDeadLetter(ctx, job)
        return
    }

    // Exponential backoff: 30s, 2m, 8m, 30m, 2h
    backoffDurations := []time.Duration{
        30 * time.Second,
        2 * time.Minute,
        8 * time.Minute,
        30 * time.Minute,
        2 * time.Hour,
    }
    delay := backoffDurations[job.RetryCount-1]
    job.RetryAfter = time.Now().Add(delay)

    // Schedule retry using a sorted set as a delay queue
    payload, _ := json.Marshal(job)
    r.RedisClient.ZAdd(ctx, queue+":delayed", &redis.Z{
        Score:  float64(job.RetryAfter.Unix()),
        Member: string(payload),
    })
}

// Separate goroutine promotes delayed jobs back to the main queue
func (r *RedisQueue) promoteDelayedJobs(ctx context.Context, queue string) {
    ticker := time.NewTicker(5 * time.Second)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            now := float64(time.Now().Unix())
            jobs, err := r.RedisClient.ZRangeByScore(ctx, queue+":delayed", &redis.ZRangeBy{
                Min: "-inf",
                Max: fmt.Sprintf("%f", now),
            }).Result()
            if err != nil {
                continue
            }

            for _, jobPayload := range jobs {
                // Atomically move from delayed set to main queue
                r.RedisClient.ZRem(ctx, queue+":delayed", jobPayload)
                r.RedisClient.RPush(ctx, queue, jobPayload)
            }
        }
    }
}

The retry mechanism uses a Redis sorted set as a delay queue. Failed jobs are added to the sorted set with their RetryAfter timestamp as the score. A background goroutine polls the sorted set every 5 seconds, finds all jobs whose score (retry time) is in the past, and promotes them back to the main BLPop queue. This gives us exponential backoff without blocking the main consumer loop.

The backoff schedule (30s, 2m, 8m, 30m, 2h) was tuned for webhook delivery. Most transient failures (target server overloaded, temporary DNS issues) resolve within minutes. The escalating delays prevent hammering a struggling endpoint while still ensuring timely delivery once it recovers. After 5 retries spanning approximately 2.5 hours, the job moves to the dead letter queue.

Comparing the Three Patterns

Here is how the three patterns compare across key dimensions:

                    | Streams + XREADGROUP    | Spring Listener         | BLPop
--------------------|-------------------------|-------------------------|------------------
Delivery guarantee  | At-least-once           | At-least-once           | At-most-once*
Consumer groups     | Native                  | Native (via Spring)     | None (1 consumer/queue)
Message ordering    | Per-stream FIFO         | Per-stream FIFO         | Strict FIFO
Stale msg recovery  | XAUTOCLAIM              | XPENDING + XCLAIM       | N/A (msg is gone)
Dead letter         | Manual (DLQ stream)     | Manual (scheduled)      | Manual (sorted set)
Complexity          | Medium                  | Low (framework managed) | Low
Message visibility  | Full (XRANGE, XINFO)    | Full                    | None after pop
Replay capability   | Yes (reset group pos)   | Yes                     | No

* BLPop is at-most-once from Redis's perspective. We achieve
  at-least-once via the PostgreSQL reconciliation job.

Use Streams + XREADGROUP when you need multiple consumers processing the same stream in parallel (consumer groups), when message visibility and replay are important, and when you want Redis to handle the delivery tracking. This is the right choice for our document processing pipeline where jobs are long-running (3-8 seconds), failures must be retried automatically, and we need to inspect the pending state during debugging.

Use Spring StreamMessageListenerContainer when you are in the Java/Spring ecosystem and want the framework to manage the polling loop, threading, and error handling. It reduces boilerplate significantly at the cost of some flexibility (no native XAUTOCLAIM support, less control over the read loop). This is our choice for engage-flow because the team is Java-centric and the Spring abstraction aligns with the existing codebase patterns.

Use BLPop when simplicity is paramount, when you have a separate durability mechanism (like our PostgreSQL reconciliation), and when consumer groups are not needed. The connector-service uses a single consumer per queue, and the at-most-once semantics are acceptable because the reconciliation job covers the gap. BLPop's biggest advantage is that it is trivial to understand: push on one end, pop on the other. There is no consumer group state, no PEL, no claiming -- just a list.

Monitoring Stream Lag

Regardless of which pattern you use, monitoring is essential. We scrape the following metrics from Redis every 30 seconds via a Prometheus exporter:

// For Redis Streams (VKYC, engage-flow)
func (m *Monitor) collectStreamMetrics(ctx context.Context, streamKey string, group string) {
    // Stream length: total messages in the stream
    length, _ := m.redis.XLen(ctx, streamKey).Result()
    m.gauge.Set("redis_stream_length", float64(length), streamKey)

    // Pending count: unacknowledged messages in the consumer group
    pending, _ := m.redis.XPending(ctx, streamKey, group).Result()
    m.gauge.Set("redis_stream_pending", float64(pending.Count), streamKey, group)

    // Per-consumer pending and idle time
    consumers, _ := m.redis.XInfoConsumers(ctx, streamKey, group).Result()
    for _, c := range consumers {
        m.gauge.Set("redis_consumer_pending", float64(c.Pending), streamKey, group, c.Name)
        m.gauge.Set("redis_consumer_idle_ms", float64(c.Idle), streamKey, group, c.Name)
    }

    // DLQ length
    dlqLen, _ := m.redis.XLen(ctx, streamKey+":dlq").Result()
    m.gauge.Set("redis_stream_dlq_length", float64(dlqLen), streamKey)
}

// For BLPop queues (connector-service)
func (m *Monitor) collectQueueMetrics(ctx context.Context, queue string) {
    // Queue depth
    length, _ := m.redis.LLen(ctx, queue).Result()
    m.gauge.Set("redis_queue_length", float64(length), queue)

    // Delayed retry queue depth
    delayedLen, _ := m.redis.ZCard(ctx, queue+":delayed").Result()
    m.gauge.Set("redis_queue_delayed", float64(delayedLen), queue)
}

These metrics feed into Grafana dashboards with the following alert rules:

Stream length > 10,000: Consumers are falling behind. This triggers a warning alert. At 50,000, it becomes critical and pages on-call.

Consumer idle > 5 minutes: A consumer has stopped reading. This usually means the worker process crashed or is stuck in a deadlock. Critical alert.

Pending count > 1,000: Messages are being delivered but not acknowledged, indicating processing failures. Warning alert.

DLQ length > 50: Systematic failure pattern. This typically means an upstream dependency is down (OCR service unreachable, partner webhook endpoint changed) rather than individual bad messages. Critical alert with a link to the admin DLQ viewer.

BLPop queue length > 5,000: Webhook delivery is backed up. This usually indicates a slow partner endpoint causing the consumer to block on HTTP calls rather than processing jobs. Warning alert.

Production Numbers

Across all three services, the daily job volume is approximately 35,000 jobs: ~8,000 document processing jobs (VKYC), ~12,000 workflow events (engage-flow), and ~15,000 webhook deliveries (connector-service). The Redis instance is a single r6g.large (2 vCPU, 16GB) running Redis 7.0 with AOF persistence (everysec fsync). Memory usage for all streams and queues combined is under 200MB.

The document processing stream (VKYC) has a median processing time of 4.2 seconds per job and a P99 of 11 seconds. The DLQ rate is 0.3% -- mostly caused by corrupt image uploads that the OCR service cannot parse. The webhook delivery queue (connector-service) has a median delivery time of 380ms and a first-attempt success rate of 96.4%. The remaining 3.6% are retried successfully within 2 minutes, and less than 0.1% ever reach the DLQ.

Redis Streams will not replace Kafka for high-throughput event streaming at millions of messages per second. But for the 10K-100K jobs/day range with a Redis instance you already operate, they provide a remarkably capable distributed job processing system. And for even simpler needs, BLPop with a sorted-set delay queue gives you a production-grade retry system in under 100 lines of code.
