Dynamic Mock Services: Testing Without Dependencies

Every microservices team hits the same wall: you cannot test your service because the three services it depends on are down, broken, or rate-limited in the staging environment. Our platform integrates with over 20 external services -- payment gateways, Aadhaar verification APIs, credit bureaus, SMS providers, NSDL PAN validation, bank account verification, and more. Each has its own sandbox environment with its own reliability problems. On any given day, at least two sandboxes would be down, blocking QA entirely. Teams resorted to hardcoding mock responses in their application code, scattered across feature branches, never cleaned up, and a constant source of production incidents when someone forgot to remove a if (env == "staging") return fakePANResponse().

We evaluated existing solutions: WireMock, MockServer, Hoverfly, and Mountebank. WireMock came closest to our needs with its request matching and response templating, but it fell short in three critical areas. First, it has no concept of stateful sequences where the response to the same request changes based on prior interactions (essential for payment flows). Second, its proxy mode is all-or-nothing rather than policy-driven (we needed to mock specific scenarios while proxying everything else to the real service simultaneously). Third, configuring it requires JSON stubs that developers must maintain in version control -- we needed a UI that QA engineers could operate without touching code. So we built our own: Simulation, a universal mock service platform with a policy engine, stateful sequences, transparent proxying, and a React UI bundled into a single Spring Boot JAR.

The MockPolicy Data Model

The foundation of the system is the MockPolicy entity. Every mock behaviour is expressed as a policy with match conditions and a response definition:

@Entity
@Table(name = "mock_policies")
public class MockPolicy {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String serviceName;       // e.g., "aadhaar-verification"
    private String method;            // GET, POST, PUT, etc.
    private String pathPattern;       // /api/v1/verify, supports wildcards

    private Integer priority;         // Lower number = higher priority
    private Boolean active;

    @Enumerated(EnumType.STRING)
    private PolicyType type;          // STATIC, SEQUENCE, PROXY

    @Column(columnDefinition = "jsonb")
    private String conditions;        // JSON array of match conditions

    @Column(columnDefinition = "jsonb")
    private String responseConfig;    // Response template, status, headers

    private Integer latencyMinMs;     // Minimum simulated latency
    private Integer latencyMaxMs;     // Maximum simulated latency
    private Double failureRate;       // 0.0 to 1.0, percentage of requests that get error response

    @Column(columnDefinition = "jsonb")
    private String failureResponse;   // Response to return on simulated failure

    private String proxyUpstreamUrl;  // For PROXY type policies

    @Column(columnDefinition = "jsonb")
    private String sequenceConfig;    // For SEQUENCE type: ordered list of responses

    private Long sequenceTtlSeconds;  // Auto-reset sequence state after this duration
    private String sequenceKeyField;  // JSON path to extract the state key (e.g., "$.orderId")
}

The conditions field stores a JSON array of match conditions with AND/OR composition. The responseConfig stores the response template including HTTP status, headers, and body with dynamic placeholders. The sequenceConfig stores an ordered array of responses for stateful flows. This single entity covers all three mock modes: static responses, stateful sequences, and transparent proxying. PostgreSQL's JSONB columns give us flexible schema for the configuration data while keeping the relational structure for queries like "find all active policies for service X ordered by priority."

The Policy Evaluation Engine

When an incoming request hits the mock service, the policy engine evaluates all active policies for the matching service and path, ordered by priority. Here is the core condition evaluation logic from the Simulation codebase:

private boolean checkCondition(List<SeqCondition> seqConditions, Object source) {
    if (seqConditions == null || seqConditions.isEmpty()) {
        return true; // No conditions = always match
    }

    for (SeqCondition condition : seqConditions) {
        String actualValue = extractValue(source, condition.getField());
        String expectedValue = condition.getValue();

        boolean matched = switch (condition.getOperation()) {
            case EQUALS ->
                actualValue != null && actualValue.equals(expectedValue);
            case CONTAINS ->
                actualValue != null && actualValue.contains(expectedValue);
            case REGEX ->
                actualValue != null && Pattern.compile(expectedValue).matcher(actualValue).matches();
            case EXISTS ->
                actualValue != null;
            case NOT_EXISTS ->
                actualValue == null;
            case GREATER_THAN ->
                actualValue != null && Double.parseDouble(actualValue) > Double.parseDouble(expectedValue);
        };

        if (condition.isNegate()) {
            matched = !matched;
        }

        if (!matched) {
            return false; // AND logic: all conditions must match
        }
    }
    return true;
}

The six operations cover the vast majority of matching scenarios we encounter in production. EQUALS handles exact matches on headers, query parameters, and JSON body fields -- used for routing by API key, endpoint, or specific identifiers. CONTAINS is the substring match we use for matching partial request bodies, like any Aadhaar verification request containing a specific state code in the address field. REGEX gives full pattern matching for PAN card formats ([A-Z]{5}[0-9]{4}[A-Z]), phone numbers, or complex XML structures in SOAP requests from legacy bank integrations. EXISTS and NOT_EXISTS check field presence regardless of value -- essential for routing based on optional headers like X-Idempotency-Key or X-Request-Priority. GREATER_THAN was added for amount-based routing: mock a timeout for transactions above 100,000 INR while returning success for smaller amounts.

The extractValue method supports dot-notation JSON paths (body.customer.address.state), header extraction (header.X-Api-Key), query parameter extraction (query.version), and path variable extraction (path.0 for the first path segment). The negate flag on each condition allows inverse matching without needing separate NOT_EQUALS, NOT_CONTAINS, etc. operations, keeping the operation set minimal.

All conditions within a single policy are combined with AND logic (every condition must match). For OR logic, you create multiple policies with the same response -- the priority system ensures the first matching policy wins. This keeps the evaluation model simple and predictable: policies are tried in order, and the first complete match produces the response.

Stateful Sequence Management

Some APIs are inherently stateful. A payment flow requires: create order (returns orderId with status "created"), check status (returns "processing"), check again (returns "completed"). A credit bureau check returns "initiated" on first call, then "report_ready" on subsequent polls. A simple request-response mock cannot model these multi-step interactions.

Our sequence management uses a ConcurrentHashMap keyed by a configurable field extracted from the request:

@Component
public class SequenceStateManager {

    private final ConcurrentHashMap<String, SequenceState> stateMap
        = new ConcurrentHashMap<>();

    public MockResponse getNextResponse(MockPolicy policy, Object requestBody) {
        String keyField = policy.getSequenceKeyField(); // e.g., "$.orderId"
        String keyValue = JsonPath.read(requestBody, keyField).toString();
        String stateKey = policy.getId() + ":" + keyValue;

        SequenceState state = stateMap.compute(stateKey, (k, existing) -> {
            if (existing == null || existing.isExpired()) {
                return new SequenceState(0, Instant.now(), policy.getSequenceTtlSeconds());
            }
            int nextIndex = Math.min(existing.getCurrentIndex() + 1,
                policy.getSequenceResponses().size() - 1);
            return new SequenceState(nextIndex, existing.getCreatedAt(),
                policy.getSequenceTtlSeconds());
        });

        return policy.getSequenceResponses().get(state.getCurrentIndex());
    }

    @Scheduled(fixedRate = 60_000)
    public void cleanupExpiredStates() {
        stateMap.entrySet().removeIf(e -> e.getValue().isExpired());
    }
}

The state key is a composite of policyId:extractedKeyValue. This means multiple concurrent test flows using different order IDs maintain independent sequence positions. When QA Engineer A creates an order with ID "order-123" and QA Engineer B creates one with "order-456", their respective status check sequences advance independently.

The ConcurrentHashMap.compute method provides atomic read-modify-write semantics. If no state exists for the key (first request in the sequence), a new state is created at index 0. If state exists, the index advances to the next response. The Math.min clamp prevents index-out-of-bounds: once the sequence reaches the last response, it stays there for all subsequent requests. This models the real-world behaviour where a payment status eventually settles on "completed" and stays there.

The TTL-based expiry (configured per policy, typically 5-30 minutes) ensures that abandoned test flows do not leak state indefinitely. The scheduled cleanup runs every 60 seconds and removes expired entries. In production, the state map rarely exceeds a few hundred entries since most test flows complete within minutes.

Dynamic Placeholder Resolution

Static mock responses get stale fast. If your Aadhaar verification mock always returns the same name regardless of input, your UI tests cannot verify that the correct data flows through the system. Our mock responses support dynamic placeholders that map request fields to response fields at runtime:

public class PlaceholderResolver {

    private static final Pattern PLACEHOLDER_PATTERN =
        Pattern.compile("\\{\\{(.+?)}}");

    public String resolve(String template, HttpServletRequest request, Object body) {
        return PLACEHOLDER_PATTERN.matcher(template).replaceAll(match -> {
            String expression = match.group(1).trim();

            // Request field references
            if (expression.startsWith("request.body.")) {
                String jsonPath = "$." + expression.substring("request.body.".length());
                return JsonPath.read(body, jsonPath).toString();
            }
            if (expression.startsWith("request.header.")) {
                return request.getHeader(expression.substring("request.header.".length()));
            }
            if (expression.startsWith("request.query.")) {
                return request.getParameter(expression.substring("request.query.".length()));
            }

            // Helper functions
            if (expression.startsWith("randomInt")) {
                String[] parts = expression.split("\\s+");
                int min = Integer.parseInt(parts[1]);
                int max = Integer.parseInt(parts[2]);
                return String.valueOf(ThreadLocalRandom.current().nextInt(min, max + 1));
            }
            if (expression.equals("timestamp")) {
                return Instant.now().toString();
            }
            if (expression.equals("uuid")) {
                return UUID.randomUUID().toString();
            }
            if (expression.startsWith("dateOffset")) {
                String[] parts = expression.split("\\s+");
                int days = Integer.parseInt(parts[1]);
                return LocalDate.now().plusDays(days).toString();
            }

            return match.group(0); // Unknown placeholder, leave as-is
        });
    }
}

A mock response template for a PAN verification API might look like:

{
    "status": "success",
    "pan": "{{request.body.panNumber}}",
    "name": "{{request.body.fullName}}",
    "verificationId": "{{uuid}}",
    "verifiedAt": "{{timestamp}}",
    "validUntil": "{{dateOffset 365}}",
    "referenceNumber": "PAN-{{randomInt 100000 999999}}"
}

When a request arrives with {"panNumber": "ABCDE1234F", "fullName": "Rahul Sharma"}, the resolved response contains the actual PAN number and name from the request, a fresh UUID, the current timestamp, a date one year from now, and a random 6-digit reference number. This makes mock responses realistic enough for full end-to-end testing, including UI validation that the correct data flows through the entire pipeline.

Transparent Proxy Mode

The transparent proxy is the feature that made adoption across teams effortless. When a policy has type PROXY, or when no policy matches and the service has a default upstream configured, the request is forwarded to the real service:

public ResponseEntity<String> proxyRequest(HttpServletRequest request,
                                              String upstreamUrl,
                                              byte[] body) {
    // Build upstream request preserving method, headers, and body
    HttpHeaders headers = extractForwardHeaders(request);
    headers.set("X-Forwarded-For", request.getRemoteAddr());
    headers.set("X-Mock-Proxy", "simulation");

    RequestEntity<byte[]> upstream = new RequestEntity<>(
        body, headers,
        HttpMethod.valueOf(request.getMethod()),
        URI.create(upstreamUrl + request.getRequestURI()
            + (request.getQueryString() != null ? "?" + request.getQueryString() : ""))
    );

    try {
        ResponseEntity<String> response = restTemplate.exchange(upstream, String.class);

        // Log the real response for potential mock creation
        auditLog.logProxyResponse(request, response);

        return ResponseEntity
            .status(response.getStatusCode())
            .headers(response.getHeaders())
            .body(response.getBody());
    } catch (HttpClientErrorException | HttpServerErrorException e) {
        auditLog.logProxyError(request, e);
        return ResponseEntity
            .status(e.getStatusCode())
            .body(e.getResponseBodyAsString());
    } catch (ResourceAccessException e) {
        auditLog.logProxyTimeout(request, e);
        return ResponseEntity
            .status(HttpStatus.GATEWAY_TIMEOUT)
            .body("{\"error\": \"upstream_timeout\", \"message\": \"" + e.getMessage() + "\"}");
    }
}

The proxy adds X-Forwarded-For for upstream logging and X-Mock-Proxy: simulation so the upstream service can distinguish proxied requests from direct calls if needed. The extractForwardHeaders method copies all request headers except Host (which must reflect the upstream, not the mock service) and hop-by-hop headers like Connection and Transfer-Encoding.

The audit log captures both the request and the real upstream response. From the Simulation UI, QA engineers can browse these logged interactions and convert any real response into a mock policy with a single click. This workflow is the fastest path from "I need to mock this specific scenario" to "it is mocked": start the proxy, trigger the real flow, find the response you want in the log, click "Create Mock From Response," adjust the conditions, and save. No JSON editing, no file commits, no redeployments.

Latency Simulation and Failure Injection

A mock that responds in 1ms does not tell you how your service behaves when the real API takes 800ms. Each policy can be configured with latency bounds and a failure rate:

private void applyLatencyAndFailure(MockPolicy policy, HttpServletResponse response)
        throws IOException, InterruptedException {

    // Simulate latency
    if (policy.getLatencyMinMs() != null && policy.getLatencyMaxMs() != null) {
        int delay = ThreadLocalRandom.current()
            .nextInt(policy.getLatencyMinMs(), policy.getLatencyMaxMs() + 1);
        Thread.sleep(delay);
    }

    // Simulate failures
    if (policy.getFailureRate() != null && policy.getFailureRate() > 0) {
        if (ThreadLocalRandom.current().nextDouble() < policy.getFailureRate()) {
            String failureBody = policy.getFailureResponse() != null
                ? policy.getFailureResponse()
                : "{\"error\": \"simulated_failure\"}";
            response.setStatus(HttpStatus.INTERNAL_SERVER_ERROR.value());
            response.getWriter().write(failureBody);
            return;
        }
    }
}

Setting latencyMinMs: 200, latencyMaxMs: 800 produces a uniform random delay between 200ms and 800ms on every request. Setting failureRate: 0.1 causes 10% of requests to return a 500 error with the configured failure response body. This combination has caught multiple production-critical bugs: a payment service that did not implement retry logic for transient failures, an SMS provider integration with a 3-second timeout that was too aggressive for the real API's P95 latency, and a circuit breaker that was configured with a threshold too low for the upstream's normal error rate.

The Unified JAR Build

The entire platform deploys as a single executable JAR. The React frontend is built during the Maven package phase and its output is copied into the Spring Boot static resources directory:

<!-- pom.xml excerpt -->
<plugin>
    <groupId>com.github.eirslett</groupId>
    <artifactId>frontend-maven-plugin</artifactId>
    <version>1.15.0</version>
    <configuration>
        <workingDirectory>src/main/frontend</workingDirectory>
        <nodeVersion>v20.11.0</nodeVersion>
    </configuration>
    <executions>
        <execution>
            <id>install-node-npm</id>
            <goals><goal>install-node-and-npm</goal></goals>
        </execution>
        <execution>
            <id>npm-install</id>
            <goals><goal>npm</goal></goals>
            <configuration><arguments>ci</arguments></configuration>
        </execution>
        <execution>
            <id>npm-build</id>
            <goals><goal>npm</goal></goals>
            <configuration><arguments>run build</arguments></configuration>
        </execution>
    </executions>
</plugin>

<plugin>
    <artifactId>maven-resources-plugin</artifactId>
    <executions>
        <execution>
            <id>copy-frontend-build</id>
            <phase>generate-resources</phase>
            <goals><goal>copy-resources</goal></goals>
            <configuration>
                <outputDirectory>${project.build.directory}/classes/static</outputDirectory>
                <resources>
                    <resource>
                        <directory>src/main/frontend/build</directory>
                    </resource>
                </resources>
            </configuration>
        </execution>
    </executions>
</plugin>

The frontend-maven-plugin downloads Node.js and npm during the build, runs npm ci for deterministic dependency installation, and executes the React build. The maven-resources-plugin copies the build output to Spring Boot's static directory. The result is a single simulation.jar that serves both the React SPA (at the root path) and the mock API (under /api). No separate frontend deployment, no CORS configuration, no nginx reverse proxy, no Docker compose with multiple containers. Teams download the JAR, run java -jar simulation.jar, and have a fully functional mock service with UI at http://localhost:8080.

CodePipeline Deployment

The Simulation service runs on ECS Fargate behind an internal ALB. The CI/CD pipeline is AWS CodePipeline with three stages:

# buildspec.yml
version: 0.2
phases:
  install:
    runtime-versions:
      java: corretto17
  build:
    commands:
      - mvn clean package -DskipTests
      - docker build -t simulation:${CODEBUILD_RESOLVED_SOURCE_VERSION} .
      - aws ecr get-login-password | docker login --username AWS --password-stdin $ECR_REPO
      - docker tag simulation:${CODEBUILD_RESOLVED_SOURCE_VERSION} $ECR_REPO:latest
      - docker push $ECR_REPO:latest
  post_build:
    commands:
      - printf '[{"name":"simulation","imageUri":"%s:latest"}]' $ECR_REPO > imagedefinitions.json
artifacts:
  files:
    - imagedefinitions.json

CodePipeline watches the main branch, triggers CodeBuild to package the JAR and build the Docker image, pushes to ECR, and then ECS performs a rolling update with the new image. The entire pipeline from git push to live deployment takes about 4 minutes. The Fargate task runs with 1 vCPU and 2GB memory, which handles our staging environment load (approximately 50 configured services with 200+ policies) comfortably.

Real-World Impact

Since deploying Simulation, our QA team's dependency-blocked time dropped from an average of 3.2 hours per sprint to effectively zero. The platform currently mocks 23 external services with 340 active policies across 6 teams. The most complex service (a bank's payment gateway) has 28 policies covering success flows, various decline codes, timeout scenarios, partial failures, and a 5-step sequence that models the complete payment lifecycle from initiation through settlement.

The investment paid for itself within the first month when we caught a payment retry bug: our service was retrying on HTTP 409 Conflict (which the payment gateway returns for duplicate idempotency keys), causing double charges. This bug was invisible in the real sandbox because the sandbox never returned 409. With Simulation, we added a policy that returns 409 when the X-Idempotency-Key header matches a previously seen value, and the bug surfaced immediately.
